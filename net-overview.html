CLR - Common Language Runtime
The C# Compiler compiles C# codes to Intermediate Language Code, also known as IL code. IL codes is run by the common language runtime (CLR).
What does CLR do?
provides a JIT compiler to compile IL codes
provides a type system to load types from assemblies
garbage collection
manage threads

IL code is similar to assembly code, but has more object-oriented features
When we run IL codes in CLR, it first compiles the IL codes with a JIT compiler(RyuJIT) into any platform-specific native code. The term JIT simply means the compilation of IL codes is delayed to be complete in runtime. The JIT compiler is not only faster than the C# compiler, but it also has better support for using Edit & Continue while you're debugging the application with Visual Studio.
The runtime also includes a type system with a type loader that is responsible for loading types from assemblies. Security infrastructure with the type system verifies whether certain type system structures are permittedâ€”for example, with inheritance.
After instances of types are created, they also need to be destroyed, and memory needs to be recycled. Another feature of the runtime is the garbage collector. The garbage collector cleans up memory from objects that are no longer referenced in the managed heap.
The runtime is also responsible for threading. When you are creating a managed thread from C#, it is not necessarily a thread from the underlying operating system. Threads are virtualized and managed by the runtime.
.NET Compiler Platform
Roslyn is the open-source implementation of both the C# and Visual Basic compilers. It allows you to interact with the compilation process, work with syntax trees, and access the semantic model that is defined by language rules. You can use Roslyn to write code analyzers and refactoring features. You also can use Roslyn with a new feature of C# 9, code generators.

.NET Standard
NET Standard is a formal specification of .NET APIs that are available on multiple .NET implementations. The motivation behind .NET Standard was to establish greater uniformity in the .NET ecosystem. .NET 5 and later versions adopt a different approach to establishing uniformity that eliminates the need for .NET Standard in most scenarios.
https://docs.microsoft.com/en-us/dotnet/standard/net-standard?tabs=net-standard-2-1
