Overview
C#不算是一个纯粹的面向对象的程序语言，因为它提供了多种编程范式(programming paradigms)。但是，Object Orientation是C#中的一个重要概念，也是.Net所提供的所有libraries所遵循的核心设计理念。
OOP中三个最重要的概念就是: inheritance, encapsulation, polymorphism, 在这一章中我们分别对这三个概念进行阐述。
Encapsulation
Encapsulation in C# is implemented with different levels of access to object data that can be specified using the following access modifiers:
Public: Access to all code in the program
Private: Access to only members of the same class
Protected: Access to members of same class and its derived classes
Internal: Access to current assembly
Protected Internal: Access to current assembly and types derived from containing class
C#中的 property 则是另一种封装类中私有数据的方式，关于 property 在上一章中有详细描述。

Polymorphism
与C家族的大多数语言一样，C#中的多态是通过虚函数实现的。
What is virtual method?
With virtual methods, the decision of which method to invoke is delayed in runtime.
the compiler creates a virtual method table, or VTable, that lists the methods that can be invoked during runtime。在运行时，通过对象的类型来选择符合要求的虚函数进行调用。
出于性能考虑，c#中的成员函数不会被默认为虚函数，非虚函数则不会被添加到虚函数表中，编译器会根据函数地址直接调用。只有在函数前添加了 virtual / override 关键字才会被当做虚函数。
What is virtual Method Table?
每个类都会维护一个虚函数表，表中包含了一系列函数指针，指向每个类对应的所有虚函数。
each instance of a class has a pointer to this table, which is used when you call a virtual method from the instance. This is because a call to a virtual method should call the method associated with the class of the actual object, not the class of the reference to the object
当一个类没有任何虚函数时，C# compiler会为我们将这张表自动优化掉。而在有虚函数的类的实例中，vtable会被分配到这个实例的内存中，所以，当我们用父类的指针来操作一个子类的时候，这张虚函数表就显得由为重要了，它就像一个地图一样，指明了实际所应该调用的函数。
public class Parent { public virtual void GetName() { Debug.Log("Parent"); } } public class Son { public override void GetName() { Debug.Log("son"); } } // 第一个例子中，由于a是一个Parent类的实例，因此它的内存中由一个指向Parent类虚函数表的指针 // 因此，a.GetName() 会指向parent类中定义的 GetName() 函数。 Parent a = new Parent(); a.GetName(); // 第二个例子中，由于b是一个Son类的实例，因此它的内存中由一个指向Son类虚函数表的指针 // 因此，b.GetName() 会指向Son类中重载的 GetName() 函数。 Son b = new Son(); b.GetName(); // 第三个例子中，c用一个Parent类型的指针指向Son类型的对象，因此这个对象的内存 // 中保存的仍然是一个指向Son类虚函数表的指针！ // 因此，b.GetName() 会指向Son类中重载的 GetName() 函数。 // 由此可见，虚函数的调用只与内存中的对象类型有关，与我们用什么类型的指针指向它无关 Parent c = new Son(); c.GetName();

Abstract Class and Abstract Method
Abstract class cannot be instantiated
Abstract method cannot have an implementation, and must be overridden in any nonabstract derived class.
Abstract methods are automatically virtual, and they can only be declared in an abstract class.
Sealed Classes and Methods
如果你不希望一个类有任何子类，应该为其添加sealed修饰符。
The most likely situation in which you'll mark a class or method as sealed is if the class or method is internal to the operation of the library, class, or other classes that you are writing. Overriding methods could lead to instability of the code, so you seal the class to make sure overriding is impossible.
同时，当我们将一个类盖棺定论后(sealed)，编译器就会知道这个类无法延伸出更多的子类，也就可以对它的虚函数表做一些优化和删减，由此提升一些性能。
Interfaces
C#是一个单继承语言，意味着C#类只能有一个父类。but by using interfaces, we can bring multiple inheritance into C#.
Predefined Interfaces
IDisposible
C#虽然允许程序员用与C/C++相同的语言定义类的析构函数，但在.NET中，GC线程控制着对象资源的释放，程序员无法掌控析构函数的调用时机。对于一些非托管资源，比如数据库链接对象等，需要实现IDisposable接口进行手动的垃圾回收。
public class Font: IDisposable { public void Dispose() { //... } } using(Font f = new Font()) { //... } //上面这段代码会被编译器字段转换为以下代码 Font f = new(); try { //... } finally { f.Dispose(); }

Dependency Injection with Interfaces
//Assume we want to add a method in the Shape class, to print some details //of any Shape object public class Shape_v1 { protected virtual void DisplayShape() { Debug.Log($"Shape with {this.Position} and {this.Size}"); } } //但是，上面这种实现方式会导致 DisplayShape() 函数严重依赖于 Debug 类 //to make it independent of Debug class, we can define a ILogger interface //this interface specifies the Log() method where a string can be passed as parameter public interface ILogger { void Log(string messsage); } //重写Shape类，我们可以使用依赖注入将一个ILogger实例传入Shape的构造函数中 public class Shape_v2 { ILogger logger; public Shape_v2(ILogger logger) { this.logger = logger; } public void DisplayShape() { Logger.Log($"Shape with {this.Position} and {this.Size}"); } }

with dependency injection, the responsibility is turned over. Instead of having a strong dependency with the implementation of the shape for the Console class, the responsibility for what is used is turned over outside of the Shape type. This way what is used can be specified from the outside.
Don't call us, we call you
dependency injection makes unit testing easier because dependencies can be easily replace with mock types.
Another advantage to when using dependency injection is that you can create platform-specific implementations. For example, showing a message box is different with the UWP（universal Windows Platform）, and IOS. You can use the interface IDialogService and define different implementations with different platforms.
Generics
代码重用的一种方案是利用继承（inheritance），另一种方案就是利用泛型类（generic types），当然两者并不冲突，可以同时使用。
在泛型类的定义中，可以使用where关键字对泛型的参数类型做出限制。
public class LinkedList<T>: IEnumerable<T> where T: ITitle { //... }
