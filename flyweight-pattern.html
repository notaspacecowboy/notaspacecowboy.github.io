定义
当我们需要大量生成某个类的实例时，可以将这些类的实例中那些可分享的数据提取出来，实现一个共享类，这就是享元模式的基本概念。
这个模式通过将对象的数据分为两种来解决这个问题。 第一种数据没有特定指明是哪个对象的实例，因此可以在它们间分享。 Gof称之为固有状态。数据的剩余部分则是变化状态，指每个实例独一无二的东西。
从例子切入 → 渲染一片森林
当我们想要在游戏中实时渲染出一整片森林时，我们讨论的是成千上万的树，每棵树都有上千个多边形组成，这意味着CPU需要每秒将百万级别的多边形发送到GPU中进行渲染，这实在是太过繁忙了。
设想一下，每棵树都有一系列与之相关的数据：
定义树干，树枝和树叶形状的多边形网格。
树皮和树叶的纹理。
在森林中树的位置和朝向。
大小和色彩之类的调节参数，让每棵树都看起来与众不同。
根据定义中对两种数据的分类方式，我们可以得出：多边形网格和纹理是可以在所有树的对象间分享的数据，而让每棵树显得独一无二的位置、朝向、大小、色彩等数据则是出于变化状态的数据。
声明 TreeModel类，用于存放可分享数据。
public class TreeModel { Mesh mesh; Texture bark; Texture leaves; }
C#
声明 Tree 类，用于存放非共享数据。
public class Tree { TreeModel model; //每个实例中保留一份对树的共享数据的引用 Vector3 position; double height; double thickness; Color barkTint; Color leafTint; }
C#
就目前而言，这看上去像是基础的资源共享，很难被称为一种模式。 部分原因是在这个例子中，我们可以为共享状态划出一个清晰的身份：TreeModel。
而当共享对象没有有效定义的实体时，使用这种模式就不那么明显（使用它也就越发显得精明）。 在那些情况下，这看上去是一个对象被魔术般地同时分配到了多个地方。
从例子切入 → 地形与区块
在上个例子中，每个树所处于的地形也需要在游戏中表示，这里可能有草，泥土，丘陵，湖泊，河流，以及其它任何你可以想到的地形。 我们基于区块建立地表：世界的表面被划分为由微小区块组成的巨大网格。 每个区块都由一种地形覆盖。
在这个例子中，每种地形类型都有一系列特性会影响游戏玩法：
决定了玩家能够多快地穿过它的移动开销。
表明能否用船穿过的水域标识。
用来渲染它的纹理。
因为我们游戏程序员偏执于效率，我们不会在每个区块中保存这些状态。 相反，一个通用的方式是为每种地形使用一个枚举。
public enum Terrain { Grass, Hill, River }
C#
World 则负责去管理巨大的世界地形网格。
public class World { Terrain tiles[WIDTH][HEIGHT]; int GetMovementCost(int x, int y) { switch(tiles[x][y]) { case Terrain.Grass: return 1; case Terrain.Hill: return 2; case Terrain.River: return 3; } } bool IsWater(int x, int y) { if(tiles[x][y] == Terrain.River) return true; return false; } }
C#
这样暂时行得通，但太丑了。移动开销和水域标识是区块的数据，但在这里它们散布在代码中。 更糟的是，简单地形的数据被众多方法拆开了。 如果能够将这些包裹起来就好了。毕竟，那是我们设计对象的目的。
因此，我们可以实现一个地形类 Terrain
public class Terrain { public int movementCost { get; } public bool isWater { get; } public Texture texture { get; } public Terrain(int movementCost, bool isWater, Texture texture) { this.movementCost = movementCost; this.isWater = isWater; this.texture = texture; } }
C#
但是我们不想为游戏世界中的每个tile都保存一个Terrain实例。 用享元的术语讲，每个tile的所有状态都是“固有的”或者说“上下文无关的“ - 处于地图左上角的草地，与处于地图右下角的草地，它们虽然是两个不同的tile，但它们所对应的Terrain实例中的所有字段都是完全一样的。
鉴于此，我们没有必要保存多个同种地形类型。 地面上的草区块两两无异。 我们不用tile对象枚举构成世界网格，而是用Terrain对象指针组成网格：
public class World { Terrain tiles[WIDTH][HEIGHT]; }
C#
每个相同地形的区块会指向相同的地形实例。
由于地形实例在很多地方使用，如果你想要动态分配，它们的生命周期会有点复杂。 因此，我们直接在游戏世界中存储它们。
public class World { public Terrain Grass { get; } public Terrain Hill { get; } public Terrain River { get; } public World() { Grass = new Terrain(1, false, new Texture()); Hill = new Terrain(3, false, new Texture()); River = new Terrain(2, true, new Texture()); } }
C#
性能分析
就像往常一样，优化的金科玉律是需求优先。 现代计算机硬件过于复杂，性能只是游戏的一个考虑方面。 在我这章做的测试中，享元较枚举没有什么性能上的损失。 享元实际上明显更快。但是这完全取决于内存中的事物是如何排列的。
使用享元对象不会搞到不可收拾。 它给了你面向对象的优势，而且没有产生一堆对象。 如果你创建了一个枚举，又在它上面做了很多分支跳转，考虑一下这个模式吧。 如果你担心性能，那么至少在把代码编程为难以维护的风格之前先做些性能分析。
