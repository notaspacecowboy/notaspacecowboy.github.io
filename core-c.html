Nullable Value Types
问题：值类型变量无法赋值为null C#解决方案：提供 Nullable<T>结构体，并通过语法糖（？关键词）将其当做第一公民。
使用 int, double, boolean 这些值类型变量时，我们无法将它们赋值为null，因为值类型变量永远不能为空。在大多数时候这都没什么问题，但是有可能会在映射到数据库或xml, json这种数据格式时出错。
这有一个Java的例子，java.util.Date类是一个引用类型，因此这个类型可以被置为null. 但是，在C#中，System.DateTime是一个值类型，DateTime变量是永远不能为空的。如果一个应用程序使用Java来写的并且打算与C#运行的一个Webservice进行映射，那么这里就会有一个问题了。如果Java的应用程序发送一个null因为CLR没有办法呈现并且对它进行操作。
使用reference type variable当然可以，但会带来额外的性能开销（overhead），引用类型的对象会被存储在堆内存中，从而需要进行GC。C# 2 中为我们引入了 nullable value type 来解决这个问题。
举个例子，对于如下代码：
int? x1 = null;

C#
the compiler sees it and changes it to use the Nullable<T> type:
Nullable<int> x1 = null;

Nullable Reference Types
问题：NullableReferenceException在C#开发中太容易发生。 C#解决方案：提供 Nullable reference type，让编译器捕捉空引用异常并在编译时报错。
作为C#程序员，最常遇到的异常类型就是NullReferenceException，即空引用报错，表示我们正在尝试调用某个空引用中的成员变量或成员函数。
NullReferenceException是C#中最常见也最容易发生的异常，从C#语言的设计理念来说，我们在每次使用reference type variable时，都应该先检查它是否为空，但从编程实现的角度来说，忘记的可能性实在太高也太过繁琐。如果compiler能捕捉空引用异常并在编译时报错，那将会非常有用。这就是Nullable Reference types被设计的原因。
设计目的
减少NullReferenceException出现的次数，这是C#中最常见的问题。
优点
人会犯错，但机器不会犯错，对reference type variable的维护职责从developer转移到compiler，大大增加了代码的安全性。
程序员不需要花时间在程序的各个地方插入空值检测，增强了代码的可读性和开发效率。

Predefined C# Types
C# 为我们提供了一系列常见的关键字，如 int, string, boolean 等等，作为语言中的内置类型。实际上，C#内部维护了一个此类关键词到内置结构体的映射表。举个例子，每当我们使用 int 关键字声明一个整形变量时，.NET编译器会识别并将其转译为一个 .NET predefined structure: System.Int32。
Integers
C#为我们提供了10种整型（如果将char也看作整型的话就是9种），分别是 sbyte/byte, short/ushort, int/uint, long/ulong, nint/nuint。每个关键字都可以映射为一个C#内置结构体。
与C++不同的是，C#中内置类型所占用的内存大小取决于应用是32位还是64位。在C#9中引入了 nint/nuint，关键词中的n代表了native，当使用这两种整型时，不管应用是32位还是64位都会占用64 bit的内存空间。
Digit Seperators
为了代码的可读性，C#允许在数字之间添加下划线，这些下划线会被编译器直接忽略。

实现该接口的类型可以通过 IComparable.CompareTo 实现两两之间的比较，经常被用于 Collection 的排序算法中。

实现该接口的类型可以通过接口中的函数将该类型变量转换为CLR类型。
CLR类型包括： Boolean、SByte、Byte、Int16、UInt16、Int32、UInt32、Int64、UInt64、Single、Double、Decimal、DateTime、Char 和 String。
C#中一个很好用的函数是Convert.ChangeType，它允许用户将某个类型转换成其他类型。但是如何你需要转换的对象不是继承自IConvertible接口，那么系统会抛出异常，转换就失败了。

Big Integer
C#提供了BigInteger类型，用于存放特别大的整型数据。这个接口本身没有内存空间的占用限制，其所占用的内存空间可以无上限的增长，直到C#无法为我们的应用melloc更多内存。
此结构体没有对应的C# keyword。由于其大小无上限的特性，BigInteger也没有所有整型都有的 MinValue/MaxValue 属性。

char[] vs String vs StringBuilder
要理解C#为什么为我们提供StringBuilder类，我们首先需要理解string的特性。
C#中的string是一个引用类型，但与其他引用类型不同的是，它的值不可变（Immutable）。string中提供的所有函数都不会修改自身的值，而是会返回一个全新的string
why we need StringBuilder class?
with many strings created, we need to clean the unused objects by the garbage collector, which frees up memory in the managed heap from objects that are no longer needed.
However, this doesn't happen immediately when the reference is not used any more, it is based on certain memory limits. Therefore, the best way is to avoid allocate memory space for objects repeatedly. In the string case, it can be done by using the StringBuilder classs.
the StringBuilder uses a memory buffer and modifies this buffer as the need arises. If the size becomes too small at some point, the buffer size always doubles（这一点与Collections很像）

Why is char[] prefered over strings when storing security sensitive information like password?
很多单机游戏的外挂都是找到存储某个游戏中变量的内存地址，然后对其进行读取和修改来实现的。同理，当用户在app中输入了一些个人的私密信息，如密码、身份证号时，如果这些值没有被即时清理掉，那么用户信息就存在泄露的风险。
Strings are immutable, which means once you've created the String, there's no way you can get rid of the data before garbage collection kicks in. 反射可以解决这个问题，但是反射永远是糟糕的解决方案。
With an array of char, you can explicitly wipe the data after you're done with it. You can overwrite the array with anything you like, and the password won't be present anywhere in the system, even before garbage collection.
Why is char[] preferred over String for passwords?
These are all the reasons, one should choose a char[] array instead of String for a password. 1. Since Strings are immutable in Java, if you store the password as plain text it will be available in memory until the Garbage collector clears it, and since String is used in the String pool for reusability there is a pretty high chance that it will remain in memory for a long duration, which poses a security threat.

ValueType and Object
在C#中，ValueType是所有值类型的基类，Object类则是所有引用类型的基类，同时也是ValueType的基类。因此，Object 类是 C#中最原始、最重要的类，是所有类，结构体，enum的基类，每个 C# 类都是它的子类，它实现了每个类都必须具有的基本方法。
这里指的“所有类”，即不管是 C# 系统所提供的标准类，还是用户自行編写的类，不管是值类型还是引用类型，都是从Object类直接或间接继承而来，它是类层次结构中的顶级类，即 C# 树型类层次结构的“根”。
当編程者定义一个类时没有使用关键字 Extends 指明它的父类，则編译器认为该类从 Object 类继承而来。但 Object 类的方法所实现的功能很有限，都是由编程者在自行设计的 class 类型的类体内编写与原型完全相同的方法来覆盖它们，以实现用户所要求的有用功能。
在 Object 类中以virtual method的方式提供了 4 个常用的方法，即 Equals、GetHashCode、GetType 以及 ToString 方法。
我们也可以将任何类型的数据赋值给object对象，即使是int或者boolean这样的值类型（boxing and unboxing）。
object a = new Book(); object b = new Human();

Processor Directives
预处理指令不会在编译后的exe中被翻译成可执行指令，但是会影响编译器的行为，如按条件决定编译某段代码，对某段代码启用/禁用Nullable reference type 等等。
C#并没有C++那样的preprocessor预处理器，所有的预处理指令都是由编译器处理的。
C# preprocessor directives
Although the compiler doesn't have a separate preprocessor, the directives described in this section are processed as if there were one. You use them to help in conditional compilation. Unlike C and C++ directives, you can't use these directives to create macros. A preprocessor directive must be the only instruction on a line.
