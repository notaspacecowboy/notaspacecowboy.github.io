Nullable Value Types
问题：值类型变量无法赋值为null C#解决方案：提供 Nullable<T>结构体，并通过语法糖（？关键词）将其当做第一公民。
使用 int, double, boolean 这些值类型变量时，我们无法将它们赋值为null，因为值类型变量永远不能为空。在大多数时候这都没什么问题，但是有可能会在映射到数据库或xml, json这种数据格式时出错。
这有一个Java的例子，java.util.Date类是一个引用类型，因此这个类型可以被置为null. 但是，在C#中，System.DateTime是一个值类型，DateTime变量是永远不能为空的。如果一个应用程序使用Java来写的并且打算与C#运行的一个Webservice进行映射，那么这里就会有一个问题了。如果Java的应用程序发送一个null因为CLR没有办法呈现并且对它进行操作。
使用reference type variable当然可以，但会带来额外的性能开销（overhead），引用类型的对象会被存储在堆内存中，从而需要进行GC。C# 2 中为我们引入了 nullable value type 来解决这个问题。
举个例子，对于如下代码：
int? x1 = null;

nullable<T>是一个结构体，因此仍然是一个值类型变量，不会带来overhead等问题。这个类型的变量在内部维护了一个布尔类型的成员变量，用来记录当前变量是否被赋值为null。

Nullable Reference Types
问题：NullableReferenceException在C#开发中太容易发生。 C#解决方案：提供 Nullable reference type，让编译器捕捉空引用异常并在编译时报错。
作为C#程序员，最常遇到的异常类型就是NullReferenceException，即空引用报错，表示我们正在尝试调用某个空引用中的成员变量或成员函数。
NullReferenceException是C#中最常见也最容易发生的异常，从C#语言的设计理念来说，我们在每次使用reference type variable时，都应该先检查它是否为空，但从编程实现的角度来说，忘记的可能性实在太高也太过繁琐。如果compiler能捕捉空引用异常并在编译时报错，那将会非常有用。这就是Nullable Reference types被设计的原因。
设计目的
减少NullReferenceException出现的次数，这是C#中最常见的问题。
优点
人会犯错，但机器不会犯错，对reference type variable的维护职责从developer转移到compiler，大大增加了代码的安全性。
程序员不需要花时间在程序的各个地方插入空值检测，增强了代码的可读性和开发效率。
