Ready-to-run(R2R) Compilation
R2R is a form of ahead-of-time (AOT) compilation.
In order to improve startup performance by reducing the amount of work the JIT compiler needs to do as your application loads, R2R compilation model can be used to pre-compile the intermediate language (IL) code into target-specific native code.
R2R binaries contain both machine native code and the IL code, because IL code is still needed for some scenarios. 所以严格意义上来说，R2R仍然是AOT + JIT Compilation。
Impact of using the ReadyToRun feature
In general, the size of an assembly will grow to between two to three times larger. This increase in the physical size of the file may reduce the performance of loading the assembly from disk, and increase working set of the process.
However, by using R2R feature the number of methods compiled at runtime is typically reduced. The startup improvement discussed here applies not only to application startup, but also to the first use of any code in the application.
代码量很大的项目采用R2R编译模型会有很高的性能提升，而代码量较小的项目的性能提升会不太明显，因为 .NET runtime libraries have already been precompiled with R2R.
How Does R2R Compilation Model Choose Precompiled Assemblies?
The SDK will precompile the assemblies that are distributed with the application. For self-contained applications, this set of assemblies will include the framework. C++/CLI binaries are not eligible for ReadyToRun compilation.
How Does R2R Compilation Model Choose Precompiled Methods?
R2R编译器的目标是在AOT时间预编译尽可能多的函数，但是在以下情况下仍然需要JIT compiler的介入在runtime进行JIT编译：
Use of generic types defined in separate assemblies.
Interop with native code.
Use of hardware intrinsics that the compiler cannot prove are safe to use on a target machine.
Certain unusual IL patterns.
Dynamic method creation via reflection or LINQ.
Native AOT deployment Model
The native AOT deployment model uses an IL to native compiler. Native AOT apps don't use a Just-In-Time (JIT) compiler when the application runs. Native AOT apps can run in restricted environments where a JIT is not allowed.
与R2R编译模型不同的地方在于，R2R虽然会预编译大部分的函数和代码，但仍然支持JIT Compiler在runtime对一些无法预编译的内容进行运行时编译，而Native AOT Model则完全禁止JIT Compiler。
Native AOT applications comes with a few fundamental limitations and compatibility issues. The key limitations include:
No dynamic loading (for example, Assembly.LoadFile)
No runtime code generation (for example, System.Reflection.Emit)
No C++/CLI
No built-in COM (only applies to Windows)
Requires trimming, which has limitations
Implies compilation into a single file, which has known incompatibilities
Apps include required runtime libraries (just like self-contained apps, increasing their size, as compared to framework-dependent apps)

Overview of Mono & Xamarin
The mono framework is an open-source implementation of .NET Framework based on the open standards for the C# language and the CLR.
Xamarin - a subsidiary of Microsoft - is a company founded by the engineers who created Mono. Xamarin is the primary maintainer and commercial sponsor of Mono.
Scripting with Mono: Why we need Mono?
在古早时期，项目都是由单个编程语言开发。程序员需要在性能更优越的底层语言（C, C++）与开发速度更快但效率更低的高级语言之间仔细选择。
编程语言的运行效率由高到低大致如下：
Assembler
Compiled Static, unsafe languages: C / C++
JITed Static, safe languages: C# / Java
以此为分界线，上下的性能差距极大
JITed Dynamic, safe language: IronPython
Interpreted dynamic, safe languages: Python, Javascript, Lua…
虽然性能差距较大，但使用脚本语言可以进行快速开发，而不必像更贴近底层的语言那样，代码库的一点小改动就会引发整个程序的重新编译。

Developers have turned to scripting as a mechanism to balance the best of both worlds, the idea is simple. The engine of the application is developed in C or C++ and usually maintained and developed by the veteran members of a team, while pieces of the UI, dialogs, interaction, or non-performance critical routines are written in a higher-level scripting language.
这样的开发架构有很多好处，首先开发时我们对上层脚本语言开发者隐藏了由C/C++所编写的引擎的一些底层细节，例如资源管理、内存管理、垃圾回收等，使脚本开发者可以专注于上层功能的实现；另外，脚本开发者在修改或开发脚本代码时不会引发底层代码的重编译。
What Mono Offers?
Mono就给我们提供了这样一套脚本系统的开发环境，使开发者可以使用包括C#在内的一系列语言来扩展我们的应用。
通常意义上的脚本语言都是解释型的，在运行效率上远远低于C/C++这样经过编译后会生成machine native code的底层语言。而Mono则扩展了脚本语言的概念，所有Mono支持的编程语言都可以被AOT compiler
The Mono Runtime
the mono runtime implements ECMA Common Language Infrastructure. The mono runtime implements the virtual machine.
The Mono runtime can be used as a stand-alone process, or it can be embedded into applications
ECMA CLI是一个规范，.NET Core CLR 和 mono runtime 都是这个规范的实现。
